>> Como dije a principios de esta semana, los archivos son uno de los sistemas fuente más comunes con los que se enfrentará todos los días como ingeniero de datos. Y es posible que recibas estos archivos o accedas a ellos desde un sistema de archivos como Google Drive o un sistema de almacenamiento de objetos como sthree, o simplemente como un archivo adjunto a un correo electrónico. Si bien los archivos pueden venir de muchos lugares diferentes, el almacenamiento de objetos es sin duda el mecanismo más importante para el almacenamiento y la recuperación de archivos en su trabajo como ingeniero de datos. Object Storage trata los datos, en este caso los archivos, como objetos individuales y los almacena en una estructura plana que no se ajusta a la jerarquía tradicional del sistema de archivos. Esto significa que, aunque esté acostumbrado a almacenar archivos en una jerarquía de carpetas y subcarpetas en su equipo local, Object Storage no tiene ninguna jerarquía. Ahora, solo como nota al margen, esto de la estructura plana puede resultar confuso. Si vas a Amazon S3, por ejemplo, hay un botón Crear carpeta y puedes crear carpetas y subcarpetas a tu antojo y almacenar tus archivos sin problemas en lo que se parece mucho a un sistema de archivos jerárquico. Sin embargo, resulta que esto es solo una característica de la interfaz de usuario para mantener las cosas organizadas de una manera que resulte familiar. El mecanismo de almacenamiento actual es plano, lo que significa que, aunque parezca que tienes carpetas y subcarpetas en la interfaz de usuario, todos los archivos se almacenan en el nivel superior. Y esto es por diseño, ya que permite un acceso rápido y sencillo a todos los objetos sin preocuparse por la sobrecarga de la estructura de carpetas. De todos modos, estoy divagando. Por lo tanto, los objetos pueden ser cualquier cosa, desde archivos CSV, JSON, texto, vídeo, imagen o audio hasta datos binarios legibles por máquina. Esta versatilidad convierte a Object Storage en el repositorio perfecto para datos semiestructurados y no estructurados, lo que puede resultar útil cuando se admiten aplicaciones como la entrega de datos para entrenar modelos de aprendizaje automático. El almacenamiento de objetos desempeña un papel crucial como fuente de datos. En semanas y cursos posteriores, verá cómo el almacenamiento de objetos también se integra a lo largo de todo el ciclo de vida de la ingeniería de datos. Pero por ahora, echemos un vistazo a algunos de los componentes clave del almacenamiento de objetos. En Object Storage, a cada objeto se le asigna un identificador único universal, o UUID, que es una especie de clave. Esta clave es necesaria para acceder al objeto correspondiente y administrarlo. Cada objeto también tiene metadatos asociados, que son información adicional sobre el objeto, como la fecha de creación, el tipo de archivo o el propietario. Vale la pena señalar que después de la escritura inicial, los objetos técnicamente se vuelven inmutables y no admiten operaciones aleatorias de escritura o adición. En este sentido, un archivo de Object Storage no es como una tabla de una base de datos relacional o un documento de una base de datos no relacional que se puede actualizar o anexar. Para cambiar los datos almacenados en un objeto, debe volver a escribir el objeto completo y hacer que el uuid apunte a este nuevo objeto. Con el almacenamiento de objetos, puede habilitar el control de versiones de objetos, lo que le permite agregar metadatos a un objeto para especificar su versión. Por lo tanto, cuando actualizas un objeto, en lugar de sobrescribir el objeto anterior con el mismo uuid, puedes conservar varias versiones de ese objeto. Entonces, ¿por qué usar el almacenamiento de objetos? Bueno, Object Storage le permite almacenar archivos de varios formatos de datos sin una estructura de sistema de archivos específica. Esto elimina la complejidad asociada con las bases de datos y los sistemas de carpetas jerárquicos. En un entorno de nube, el almacenamiento de objetos puede ampliarse fácilmente para proporcionar un espacio de almacenamiento prácticamente ilimitado para cantidades masivas de datos. En términos de disponibilidad, los datos del almacenamiento de objetos en la nube suelen replicarse en varias zonas de disponibilidad, lo que significa que los datos se replican en varios centros de datos físicos que están aislados unos de otros. Esto hace que los datos sean muy duraderos y estén disponibles incluso en el caso de desastres naturales. Por ejemplo, como mencioné en el curso anterior, Amazon S3 ofrece una durabilidad de datos de 11 nueves, lo que significa que el almacenamiento de objetos en S3 puede soportar fallos simultáneos de dispositivos o centros de datos. Además, el almacenamiento de objetos suele ser más económico que otras opciones de almacenamiento, especialmente si almacenas datos a los que no necesitas acceder de forma regular. Cloud Object Storage se usa en muchas aplicaciones y es el almacenamiento subyacente para los diseños de arquitectura más nuevos, como lagos de datos y casas de lagos de datos, debido a su flexibilidad, alta escalabilidad, rentabilidad y durabilidad. El tipo de sistema de transmisión más simple que se me ocurre es un registro. De hecho, el registro ni siquiera es un sistema. Es solo un registro de información sobre eventos que puede servir para rastrear la actividad de un sistema o una aplicación. En el curso anterior, mencioné que solía ser común que los desarrolladores consideraran los datos provenientes de las aplicaciones de software como un escape o un subproducto, que no necesariamente tenían ningún valor intrínseco por sí mismos, pero eran útiles para monitorear o depurar un sistema. Los datos específicos que se consideran más comúnmente como datos de escape son los datos contenidos en los registros producidos por las aplicaciones de software. Cuando un desarrollador implementa un producto o una plataforma, como un sitio web o una aplicación móvil, lo configurará de manera que toda la actividad que se produzca dentro de la aplicación se registre en un registro. El registro puede incluir la actividad de un usuario, como un usuario que inicia sesión o navega a una página en particular. También puede incluir un registro de eventos en el back-end, como una actualización de una base de datos o un error que se generó al intentar ejecutar un procedimiento en particular. Los registros se utilizan con mayor frecuencia en la práctica como medio para supervisar el estado de los sistemas. Los ingenieros usarán los registros para activar alertas o para depurar lo que salió mal cuando se produce un error. En este sentido, los troncos pueden parecer aburridos y la caracterización de los troncos como gases de escape para aplicaciones puede parecer apropiada. Sin embargo, los registros son una fuente rica de datos que pueden ser útiles para mucho más que solo monitorear el estado de una aplicación. Como tales, pueden ser un sistema fuente importante. Absorberás datos de tu trabajo como ingeniero de datos. En esencia, un registro es una secuencia de registros que solo se pueden anexar ordenados por tiempo que captura información sobre los eventos que ocurren en los sistemas. Por ejemplo, si eres el ingeniero de datos de una empresa de comercio electrónico, los registros de tu servidor web pueden capturar datos detallados de la actividad de los usuarios que podrían usarse para respaldar el análisis posterior de los patrones de comportamiento de los usuarios. Muchos sistemas de bases de datos tendrán registros que puede usar para rastrear los cambios en el proceso de la base de datos, conocido como captura de datos de cambios o CDC, por sus siglas en inglés. Puede usar esos cambios para activar los procesos de ingestión de modo que se ejecuten en función de la llegada de nuevos datos a la base de datos. O puede ingerir datos de registro para usarlos en ciertas aplicaciones de aprendizaje automático, como la detección de anomalías, si, por ejemplo, está ingiriendo datos de registro de sistemas de seguridad. Los registros desempeñan un papel crucial en el seguimiento de lo que ocurrió en muchos de los sistemas de software anteriores con los que trabajará. Esto los convierte en una fuente de datos rica que puede respaldar casos de uso posteriores, como el análisis de datos, la resolución de problemas, la supervisión del rendimiento , las aplicaciones de aprendizaje automático y la automatización. Como mencioné anteriormente, el registro es un registro de información sobre eventos. En general, los datos que se registran primero para cada evento de un registro son la cuenta personal, de sistema o de servicio asociada al evento, como un identificador de usuario y su dirección IP. A continuación, encontrará un registro del evento que ocurrió junto con sus metadatos. Por ejemplo, un usuario agregó un producto específico a su carrito, junto con el estado de esa acción. Por último, encontrarás un registro de la marca de tiempo del evento. Los datos de registro se pueden registrar como texto simple no estructurado o en formato JSON o CSV, o incluso como datos codificados binarios. Además de los datos que describen la hora y el contenido de un evento, los registros también suelen incluir una etiqueta para clasificar el evento asignando lo que se conoce como nivel de registro a cada registro. Los niveles de registro pueden incluir etiquetas como debug, info, warn , error o fatal que permiten saber qué tipo de información contiene un registro determinado. Por ejemplo, a un registro que contenga información de actividad básica se le asignaría el nivel de registro de información. Bueno, es posible que a un registro que contenga un mensaje de error se le asigne el nivel de registro de errores o si ocurre algo más grave, como que los sistemas principales fallan y necesitan atención urgente. Esto podría incluir el nivel de registro fatal como etiqueta. Hablaremos más sobre los niveles de registro más adelante, cuando comience a crear registros en sus propias aplicaciones de canalización de datos, en lugar de supervisar sus propios sistemas. Como ingeniero de datos, es importante que comprenda cómo trabajar con los registros, sus tipos, formatos y aplicaciones. Los registros serán una fuente importante de datos para el trabajo que realice y pueden ayudarlo a solucionar problemas, supervisar el rendimiento y atender muchos casos de uso posteriores.
